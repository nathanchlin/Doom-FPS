# 末日迷宫：从零到上线的 AI 协作开发全记录

> 一个完整的 3D Roguelike 迷宫射击游戏，从概念到部署全程由 Claude Code + Superpowers 技能驱动完成。

**在线试玩**: [https://nathanchlin.github.io/Doom-FPS/](https://nathanchlin.github.io/Doom-FPS/)

---

## 项目概览

**末日迷宫**是一个基于 Three.js + TypeScript + Vite 的浏览器端 3D FPS 游戏。玩家在随机生成的迷宫中探索，开门面对战斗或宝箱，通过卡牌系统构建 Build，挑战无尽模式看能走多远。

| 技术指标 | 数据 |
|---------|------|
| 技术栈 | Three.js + TypeScript + Vite |
| 外部依赖 | 仅 `three`（零资源文件，全程序化生成） |
| 源文件数 | 17 个 TypeScript/HTML/CSS 文件 |
| 总提交数 | 36 个 |
| 开发周期 | 1 天 |
| 部署方式 | GitHub Pages + GitHub Actions |

## 开发时间线

### 第一阶段：迷宫核心玩法（从竞技场到迷宫）

**起点**：项目原本是一个简单的竞技场射击游戏——60m×60m 的方形房间里 6 个白色方块敌人。

**Superpowers 技能发挥**：

#### 1. 头脑风暴（brainstorming）

使用 `superpowers:brainstorming` 技能，通过**逐个问题**的方式把模糊想法变成清晰设计：

```
用户想法："制作一个迷宫的玩法，玩家找门，门后可能是出口/怪物/宝箱"

→ 问题1: 门的交互方式？    → 传送到独立房间（Roguelike 房间制）
→ 问题2: 迷宫布局风格？    → 经典走廊式迷宫
→ 问题3: 循环生成节奏？    → 无尽模式，看能走多远
→ 问题4: 门后概率分布？    → 硬核型（战斗65% / 宝箱20% / 出口15%）
→ 问题5: 怪物变化程度？    → 2-3种敌人类型
→ 问题6: 资源管理方式？    → 仅宝箱提供（硬核）
```

6 个问题，每个问题给出 3 个选项，用户一选就定。**没有来回讨论的低效沟通**，几分钟内把完整游戏设计锁定。

#### 2. 设计文档输出

自动生成结构化的设计文档（`docs/superpowers/specs/`），覆盖：
- 迷宫生成算法（递归回溯）
- 门与房间系统（战斗/宝箱/出口）
- 三种敌人类型（标准/冲锋/重装）
- 资源经济分析
- 游戏状态机
- HUD 变化
- 架构变更清单

#### 3. 实现计划（writing-plans）

`superpowers:writing-plans` 把设计文档转化为 **11 个粒度任务**，每个任务包含：
- 精确的文件路径
- 完整的代码实现
- 提交命令和消息
- 类型检查验证步骤

#### 4. 子代理驱动开发（subagent-driven-development）

`superpowers:subagent-driven-development` 自动派出子代理并行执行任务：

```
控制器（我）
  ├─ 子代理 A: 实现 Task 1-7（config, Maze, Level, Enemy, Door/Chest/Room, Sfx, Player）
  ├─ 子代理 B: 实现 Task 8-11（HTML/CSS, Hud, Game, 集成测试）
  └─ 类型检查 + 构建验证
```

**9 个文件的并行实现在几分钟内完成**，不是逐个手动编写。

### 第二阶段：Bug 修复（系统化调试）

迷宫系统首次运行时遇到 3 个问题：

| Bug | 根因 | 修复 |
|-----|------|------|
| 看不到门 | Level.ts 在门位置渲染了完整墙壁，遮挡了门 | 墙壁在门位置拆成两段留缺口 |
| 进门后全黑 | Player 碰撞始终用迷宫 Level，传送到房间(500,504)后被拉回迷宫 | 进房间时切换碰撞提供者为 Room |
| 仍然全黑 | `this.nearDoor` 在 async fadeIn 300ms 等待期间被游戏循环置 null | await 前用局部变量保存引用 |

`superpowers:systematic-debugging` 技能强制执行**根因调查优先**的原则——不猜测不乱试，追踪数据流找到真正原因。第三个 bug 通过让用户查看浏览器 Console 日志精确定位到 `Cannot read properties of null (reading 'roomType')` 这一行。

### 第三阶段：卡牌选择系统

用户一句话需求："进入下一层前有三张卡可选，可能是新武器、属性提升、弹匣扩容"

同样的流程：**brainstorming → 设计文档 → 实现计划 → 子代理执行**

最终实现：
- **3 种武器**：步枪（平衡）、霰弹枪（6 弹丸扩散）、狙击枪（高伤单发）
- **4 种属性卡**：生命+25、弹匣+10、速度提升、伤害×1.15
- **4 种特殊卡**：满血恢复、弹药补给、护盾（3 次半伤）、侦察（显示门类型）
- **全屏卡牌 UI**：白底黑框美术风格一致，悬停动效，点击选择

### 第四阶段：难度提升 + 走廊威胁

- **巡逻敌人**：紫色高血量怪物在迷宫走廊巡逻，发现玩家主动追击
- **红色危险区**：地板随机出现的伤害区域，持续掉血
- **视线检测**：修复敌人穿墙射击——加入射线-AABB 相交检测

### 第五阶段：汉化 + 介绍页 + 部署

- 全量中文本地化（HUD、卡牌、交互提示、死亡画面）
- 功能介绍界面（迷宫探索、战斗房间、卡牌升级、硬核生存、敌人图鉴）
- GitHub Actions 自动部署到 GitHub Pages

## Superpowers 技能体系的价值

### 1. 结构化头脑风暴（brainstorming）

**问题**：用户说"做个迷宫玩法"——这能有 100 种实现方式。

**Superpowers 做法**：
- 逐个问题引导，每次 3 个选项
- 多选题优先，减少用户认知负担
- 提出 2-3 个方案并推荐最优解
- 设计按节呈现，逐段确认

**效果**：5 分钟内从模糊想法到完整设计文档。没有反复讨论、没有遗漏需求。

### 2. 实现计划（writing-plans）

**问题**：大特性直接写代码容易丢失方向，文件间依赖混乱。

**Superpowers 做法**：
- 文件结构先行，锁定每个文件的职责
- 任务粒度到每个步骤（写代码 → 验证 → 提交）
- 完整代码在计划中，不留占位符
- 类型一致性自查

**效果**：11 个任务的迷宫系统，从计划到全部实现类型检查通过，没有返工。

### 3. 子代理并行执行（subagent-driven-development）

**问题**：11 个文件要修改，串行太慢。

**Superpowers 做法**：
- 独立任务并行派发
- 每个子代理拿到完整上下文，不需要读文件
- 实现后自动类型检查验证
- 失败自动诊断和修复

**效果**：9 个文件的并行实现 + 集成，总耗时约 10 分钟。

### 4. 系统化调试（systematic-debugging）

**问题**：3 个 bug 症状相同（黑屏），但根因完全不同。

**Superpowers 做法**：
- 强制根因调查，禁止猜测修复
- 追踪数据流：门位置 → 墙渲染 → 碰撞系统 → 异步状态
- 注入诊断日志精确定位
- 每次修复一个变量，验证后再继续

**效果**：3 个 bug 各一次修复成功，零回退。

- ✅ 资源仅宝箱获取，跨层保留

### 环境威胁
- ✅ 走廊巡逻敌人（紫色，高血量）
- ✅ 红色地板危险区（持续伤害）

- ✅ TypeScript 严格模式

- ✅ GitHub Actions 自动部署

## 总结

Superpowers 技能体系把 AI 辅助开发从"帮我写段代码"提升到了**全流程工程化**：

1. **brainstorming** 消灭了需求模糊——每个设计决策都有明确选项和推荐
2. **writing-plans** 消灭了实现混乱——每个文件、每行代码都有计划
3. **subagent-driven-development** 消灭了串行瓶颈——独立任务并行执行
4. **systematic-debugging** 消灭了盲目试错——根因驱动，一次修对

这不是"AI 帮你写代码"，这是**AI 和你一起做工程**。
